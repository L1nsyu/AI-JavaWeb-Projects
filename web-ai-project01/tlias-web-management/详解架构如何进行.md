## 一个非常标准的 Spring Boot + MyBatis 分层架构（以部门管理 Dept 为例）。

---

 _**我来为你逐层拆解，用伪代码和通俗解释告诉你每个文件里通常写什么，以及它们是怎么配合的。**_

### 一. POJO 层 (实体类 & 统一响应)
   _这里是数据的“载体”，就像送快递的盒子。_
   1. Dept (实体类)
   <br>作用：对应数据库中的 dept 表。表里有什么字段，这里就有什么属性。
   <br>实现代码样子：

    @Data // Lombok注解，自动生成get/set方法
    public class Dept {
        private Integer id;
        private String name;
        private LocalDateTime createTime;
        private LocalDateTime updateTime;
    }

   2. Result (统一响应结果)
      <br>作用：这是你和前端约定的“固定格式”。不管成功还是失败，都返回这个对象，前端好处理。
      <br>实现代码样子：

    @Data
    public class Result {
        private Integer code; // 1成功，0失败
        private String msg;   // 提示信息
        private Object data;  // 真正的数据（比如部门列表）

    // 这是一个静态方法，方便快速创建成功结果
        public static Result success(Object data) {
            Result r = new Result();
            r.code = 1;
            r.msg = "success";
            r.data = data;
            return r;
        }
    }
### 二. Mapper 层 (数据访问层)
   _这里是跟数据库“对话”的地方。_
   1. DeptMapper (接口)
      <br>注意：这是一个 Interface (接口)，不是 Class。
      <br>作用：定义 SQL 语句。你不需要写实现类，MyBatis 框架会自动帮你生成代理对象来执行 SQL。
      <br>实现代码样子：

    @Mapper // 告诉SpringBoot这是个Mapper
    public interface DeptMapper {

        // 查询所有部门
        @Select("select * from dept")
        List<Dept> list();
    
        // 根据ID删除
        @Delete("delete from dept where id = #{id}")
        void deleteById(Integer id);
    }

### 三. Service 层 (业务逻辑层)
   _这里是核心“大脑”。通常分为 接口 和 实现类。_
   1. DeptService (接口)
    <br>作用：像一份“菜单”或“合同”，只定义能做什么（有哪些功能），不写具体怎么做。
    <br>实现代码样子：

    public interface DeptService {
        // 定义一个查所有部门的方法
        List<Dept> list();
    }

   2. impl/DeptServiceImpl (实现类)
    <br>作用：真正的“厨师”，负责实现接口里的方法。
    <br>核心逻辑：它需要调用 Mapper 去拿数据，然后可能做一些逻辑处理（比如过滤、计算）。
      <br>实现代码样子：

    @Service // 必须加这个注解，把类交给Spring管理
    public class DeptServiceImpl implements DeptService {
    
        @Autowired // 注入Mapper，因为要用它查数据库
        private DeptMapper deptMapper;
        
        @Override
        public List<Dept> list() {
            // 这里可以写业务逻辑，比如记录日志等
            // 调用Mapper查数据
            return deptMapper.list();
        }
    }

### 四. Controller 层 (控制层)
   _这里是“前台接待”。_
   1. DeptController (类)
      <br>作用：接收浏览器发来的请求（比如 /depts），调用 Service，把结果封装成 Result 返回。
      <br>实现代码样子：

    @RestController // 标识这是一个Controller，且返回JSON数据
    @RequestMapping("/depts") // 定义公共路径
    public class DeptController {
    
        @Autowired // 注入Service，因为要指挥厨师干活
        private DeptService deptService;
        
        // 处理 GET 请求
        @GetMapping
        public Result list() {
            // 1. 调用Service拿到数据
            List<Dept> deptList = deptService.list();
            
            // 2. 包装成Result格式返回
            return Result.success(deptList);
        }
    }

   ## 总结：数据是怎么流动的？
   **当你（前端）发起一个请求：GET http://localhost:8080/depts**
1.    Controller (DeptController) 接到请求。
2.    Controller 喊 Service (DeptServiceImpl)：“给我查部门列表”。
3.    Service 喊 Mapper (DeptMapper)：“去数据库捞数据”。
4.    Mapper 执行 SQL select * from dept，把数据库里的行变成 POJO (Dept) 对象列表，传回给 Service。
5.    Service 把列表传回给 Controller。
6.    Controller 把列表装进 POJO (Result) 盒子里，发回给浏览器。
