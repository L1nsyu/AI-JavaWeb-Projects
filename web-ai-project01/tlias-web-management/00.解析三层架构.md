## 形象解释三层架构之间的角色是什么,以及他们之间的关系和各自的作用.
**假设你开了一家餐馆（整个 Web 项目），顾客（浏览器/前端）来吃饭。**
1. Controller 层 (控制层 / Web 层)
   * 角色：服务员 / 前台
   * 代码特征：带有 @RestController、@RequestMapping 注解。
   * 作用：
     * 接待客人：接收前端发来的请求（HTTP Request）。
     * 记菜单：接收参数（比如 ?name=zhangsan）。
     * 不亲自做饭：它只负责把任务转交给 Service (厨师)。
     * 上菜：拿到结果后，按照约定好的格式（通常是 JSON）返回给前端。
   * **原则：Controller 里的代码要少，只做“接待”和“传话”，不要写复杂的业务逻辑。**
2. Service 层 (业务层)
   * 角色：厨师
   * 代码特征：带有 @Service 注解。
   * 作用：
     * 核心大脑：这里是处理业务逻辑的地方。
     * 加工食材：比如“注册用户”，它需要检查用户名是否重复、密码加个密、计算一下积分。
     * 指挥：如果业务复杂，它可能需要调用多个 Mapper 来获取不同的数据。
     * 事务控制：如果出错了（比如钱扣了但货没发），它负责回滚（撤销操作），保证数据安全。
   * **原则：所有的“动脑子”的事情，都在这里做。**
3. Mapper / DAO 层 (持久层)
   * 角色：仓库管理员 / 采购员
   * 代码特征：带有 @Mapper 注解，通常是接口（Interface），里面写 SQL 或关联 XML。
   * 作用：
     * 只管拿货：它不关心业务逻辑，它只负责和 数据库 (仓库) 打交道。
     * 执行命令：执行增删改查（CRUD）的 SQL 语句。
     * 数据转换：把数据库里的表记录（Row）变成 Java 对象（Object），或者反过来。
     * **原则：只做最纯粹的数据库操作，不要包含复杂的判断逻辑。**
4. Entity / POJO (实体类)
   * 角色：盘子 / 食材盒 / 菜单项
   * 代码特征：带有 @Data 注解，里面只有一堆属性（字段）。
   * 作用：
   * 数据载体：它是数据的搬运工。
   * 映射：通常一个 Entity 类对应数据库里的一张表。
   * **流动：数据通过它，从数据库 -> Mapper -> Service -> Controller -> 前端，一路传过去。**

_**总结：一次完整的流程**_
1. 场景：用户点击“查询用户信息”按钮
2. 顾客 (前端) 喊了一句：“我要查 id=1 的用户！”（发起 HTTP 请求）。
3. 服务员 (Controller) 听到后，记下 id=1，转头喊 厨师 (Service)：“有人要查 1 号用户！”。
4. 厨师 (Service) 接到单子，发现只是个简单查询，不需要加工，于是喊 仓库管理员 (Mapper)：“去仓库帮我把 1 号用户的数据拿来！”。
5. 仓库管理员 (Mapper) 跑到 仓库 (数据库)，执行 SQL select * from user where id=1。
6. 仓库 (数据库) 找到数据，Mapper 把它装进 盘子 (Entity) 里，交给厨师。
7. 厨师 (Service) 确认没问题，把盘子交给服务员。
8. 服务员 (Controller) 把盘子里的数据打包成 JSON 格式，端给 顾客 (前端)。
